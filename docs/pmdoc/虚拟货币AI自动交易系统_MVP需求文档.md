## 一、项目背景

### 1.1 市场背景

加密货币市场7x24小时运行，波动剧烈。传统人工交易存在以下痛点：

- 无法24小时盯盘，错失交易机会
- 情绪化交易导致亏损
- 手忙脚乱追涨杀跌
- 技术分析耗时且门槛高

### 1.2 项目目标

构建一个**安全，透明、低门槛**的虚拟货币AI自动交易系统，让用户：

- 无需盯盘，AI自动执行交易
- 用AI辅助决策，降低情绪化交易
- 在模拟/真实环境中验证策略
- 小资金即可试跑，降低试错成本

### 3.4 风控模块（核心！）

| 维度 | 具体措施 |

| API风控 | IP白名单、请求频率限制、异常登录告警 |
| 系统风控 | 交易所连接失败自动撤单、宕机备援 |

## 四、技术架构（极简MVP版）

> 更新说明：原方案为大规模生产环境设计，MVP阶段过于复杂。现简化为「极简MVP版」，用最少技术栈最快跑通。

### 4.1 系统架构图

```
┌─────────────────────────────────────────┐
│              单体应用 (Python FastAPI)          │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌────────┐  │
│  │ 行情采集 │  │ 策略引擎 │  │ 交易执行│  │
│  │ (异步)   │  │ (同步)   │  │ (同步)  │  │
│  └─────────┘  └─────────┘  └────────┘  │
│         │              │            │       │
│         └──────────────┼────────────┘       │
│                        ▼                    │
│                 ┌───────────┐            │
│                 │  SQLite   │            │
│                 │  (本地文件) │            │
│                 └───────────┘            │
└─────────────────────────────────────────┘
```

### 4.2 技术选型

| 层级 | 技术方案 | 理由 |
|------|----------|------|
| 后端 | Python FastAPI | 一个人全栈，策略+AI+接口全能搞定 |
| 数据库 | SQLite | MVP阶段足够，零运维 |
| 定时任务 | APScheduler | 内置调度，不用额外中间件 |
| 部署 | Docker单容器 | 一键部署，简单 |
| 交易所SDK | CCXT | 统一SDK，多交易所支持 |
| 前端 | React + Ant Design | 成熟稳定，组件丰富 |
| AI能力 | OpenAI API / 本地模型 | 按需选型 |

### 4.3 核心模块

1️⃣ **数据采集模块**
   - 异步获取实时行情 (Tick/K线)
   - 交易所API对接 (CCXT)
   - 数据清洗与缓存

2️⃣ **策略引擎**
   - 策略编写框架 (回测/模拟/实盘)
   - 信号生成逻辑
   - 策略版本管理

3️⃣ **AI推理模块**
   - 预训练模型加载 / API调用
   - 实时预测 (信号评估)
   - 模型热更新

4️⃣ **风险控制模块**
   - 仓位限制
   - 止损止盈规则
   - 单日亏损上限

5️⃣ **交易执行模块**
   - 订单管理 (限价/市价)
   - 成交追踪
   - 滑点控制

6️⃣ **监控与告警**
   - 异常交易检测
   - 系统健康度检查
   - 告警通知 (飞书/邮件)

### 4.4 为什么这么简化？

| 设计点 | 理由 |
|--------|------|
| 单体架构 | MVP阶段不需要微服务，一个人能维护 |
| SQLite | 文件数据库，零运维，够用 |
| 不用Kafka | MVP数据量小，直接数据库够用 |
| Python全栈 | 一个人完成全部开发 |
| 单容器部署 | 一条docker-compose up就跑起来 |

### 4.5 方案对比

| 维度 | 原方案 | 极简MVP版（本文） | 云原生轻量版 |
|------|--------|-------------------|--------------|
| 架构 | 微服务 | 单体 | 云函数+托管服务 |
| 数据库 | PostgreSQL+InfluxDB | SQLite | Supabase |
| 消息队列 | Kafka | 无 | Redis云 |
| 部署 | K8s | Docker | Vercel/Railway |
| 开发人数 | 3-5人 | 1人 | 1-2人 |
| 上线时间 | 8周+ | 2-4周 | 4-6周 |
| 运维成本 | 高 | 极低 | 低 |
| 适合阶段 | 规模化生产 | MVP验证 | 产品增长 |

### 5.2 交互设计原则

- **极简启动**：3步完成策略启用（选策略→绑定API→设置金额）
- **透明可信**：每笔交易记录可查，策略逻辑白话文解释
- **渐进复杂度**：新手用"一键跟单"，进阶用户开放自定义参数
- **情感化反馈**：盈利弹窗、亏损预警，重要节点及时推送

## 七、安全与稳定性

### 7.1 安全保障

- **资金隔离** - 交易API仅开放交易权限，提现API禁用
- **敏感数据加密** - API Key/Secret加密存储
- **操作审计** - 所有交易指令记录日志，可追溯
- **风控多层把关** - 策略信号 → 风控审核 → 执行下单

### 7.2 稳定性保障

- **熔断机制** - API异常时自动暂停交易
- **重试机制** - 网络波动时指数退避重试
- **定时检查** - 定期检查策略运行状态
- **日志记录** - 关键操作记录日志

## 八、专家讨论要点总结

### 产品经理观点

- **定位**："让不懂编程的持币者，也能享受自动化交易带来的睡后收益"
- **亮点**：托管型安全感、小白友好、数据真实性、轻量化
- **核心**：省心，安全、可预期、低门槛

### 技术总监观点

- **架构**：单体应用 + 异步采集 + 同步执行
- **关键**：Python全栈 + SQLite + Docker
- **MVP建议**：先聚焦单一策略、模拟盘验证、最小交易对

### 交易大师观点

- **核心**："能稳住不爆仓" > "赚大钱"
- **风控**：仓位限制、强制止损、熔断冷静、禁止摊平
- **用户痛点**：本金安全 > 透明度 > 省心 > 快速见效
